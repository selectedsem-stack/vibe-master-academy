HTTP ו-APIs — איך תוכנות מדברות

1.5 שעות
רענון מ-Lesson 2

ב-Lesson 2 השתמשנו בנתונים JSON מקובץ מקומי. עכשיו נראה איך לקבל JSON מאתר אחר באינטרנט — כלומר, לקרוא API.

מה זה API באמת?

API הוא "תפריט הזמנות" של שירות אינטרנטי. הדמיון הכי קל להבין:

  • אתה (הלקוח) = קוד ה-Python שלך
  • המלצר = ה-API — מקבל את הבקשה ומחזיר תוצאה
  • המטבח = השרת של Claude / GitHub / Google
  • האוכל = ה-JSON עם המידע שביקשת

אתה לא נכנס למטבח בעצמך. אתה שולח request מסוים, ומקבל response מסוים. זה כל מה ש-HTTP הוא — שפת התקשורת בין לקוח לשרת.

GET לעומת POST

שתי ה-methods הנפוצות ביותר ב-HTTP:

Method מה הוא עושה מתי משתמשים דוגמה
GET מבקש מידע מהשרת קריאה בלבד — שליפת נתונים שלוף פרטי משתמש מ-GitHub
POST שולח מידע לשרת + מבקש תגובה יצירה / פעולה / שליחה שלח הודעה ל-Claude API
חוק האגודל

קריאה בלבד → GET. שליחת נתונים או פעולה → POST. Claude API הוא תמיד POST — אתה שולח את ההודעה שלך.

Status Codes — מה ה-API אומר לך

כל response מ-API כולל מספר תלת-ספרתי שמסביר מה קרה. חשוב ללמוד אותם בעל פה:

status_codes.txt 
200  OK              — הצלחה מלאה ✓
201  Created         — נוצר משאב חדש ✓
400  Bad Request     — הבקשה שלך שגויה (בדוק פרמטרים)
401  Unauthorized    — מפתח API שגוי או חסר
403  Forbidden       — אין לך הרשאה
404  Not Found       — ה-URL לא קיים
429  Too Many Req.   — חרגת ממגבלת קריאות (המתן)
500  Server Error    — שגיאה בצד השרת (לא אשמתך)
לזכור בקלות
2xx = הצלחה
4xx = טעות שלך
5xx = טעות של השרת

Headers — המידע שנוסע עם הבקשה

Headers הם "מטא-נתונים" שנשלחים עם כל בקשה. שני ה-headers החשובים ביותר ב-API calls:

headers_example.txt 
x-api-key: sk-ant-api03-...         (מי אתה — מפתח ה-API)
anthropic-version: 2023-06-01        (איזו גרסת API אתה משתמש)
content-type: application/json       (פורמט הנתונים שאתה שולח)
מפתח API = סיסמה

מפתח ה-API שלך הוא בדיוק כמו סיסמה. לעולם אל תכניס אותו ישירות בקוד. תמיד תשמור אותו ב-environment variable ותטען עם os.environ.get("ANTHROPIC_API_KEY"). אם בטעות עשית commit למפתח — תחליף אותו מיד.

Python requests — הקריאה הראשונה שלך

1.5 שעות

התקנת ספריית pip

requests היא ספרייה חיצונית — לא מגיעה עם Python מובנית. מתקינים עם pip:

Terminal 
pip3 install requests
פלט צפוי
Successfully installed requests-2.x.x

אחרי ההתקנה, מייבאים בתוך הקוד עם import requests.

Hello World: קריאת GET ראשונה

get_demo.py 
import requests

# קריאת GET — שולפים מידע
response = requests.get("https://api.github.com")

print(response.status_code)    # 200
print(type(response))          # <class 'requests.models.Response'>

ה-response הוא אובייקט עם שדות שימושיים:

שדה תוכן
response.status_code קוד המצב (200, 404, 500...)
response.json() ה-body כ-dict Python
response.text ה-body כ-string גולמי
response.headers כל ה-headers של התגובה

Parse JSON מ-response

parse_json.py 
import requests

response = requests.get("https://api.github.com")

# .json() ממיר את ה-body ל-dict Python
data = response.json()

# עכשיו data הוא dict רגיל — אפשר לגשת לפי key
print(data.get("current_user_url"))
# https://api.github.com/user
זיכרון מ-Lesson 2

כשעשינו json.load(f) מקובץ — קיבלנו dict Python. כשעושים response.json() מ-API — מקבלים בדיוק אותו דבר. הכלים זהים, רק המקור שונה.

קריאת POST ל-Claude API

זו הקריאה האמיתית. שים לב ל-headers ולמבנה ה-body. שים לב שהמפתח מגיע מ-environment variable, לא מהקוד עצמו.

claude_api_call.py 
import requests
import os

API_KEY = os.environ.get("ANTHROPIC_API_KEY")

response = requests.post(
    "https://api.anthropic.com/v1/messages",
    headers={
        "x-api-key": API_KEY,
        "anthropic-version": "2023-06-01",
        "content-type": "application/json"
    },
    json={
        "model": "claude-opus-4-7",
        "max_tokens": 500,
        "messages": [
            {"role": "user", "content": "סכם בעברית: Python היא שפה פופולרית."}
        ]
    }
)

if response.status_code == 200:
    data = response.json()
    print(data["content"][0]["text"])
else:
    print(f"שגיאה: {response.status_code}")

נקודות חשובות בקוד:

  • os.environ.get("ANTHROPIC_API_KEY") — מפתח מגיע מהסביבה, לא מהקוד
  • json={...} — requests ממיר אוטומטית ל-JSON ומוסיף Content-Type header
  • data["content"][0]["text"] — זה המבנה הספציפי של תגובת Claude
איך מגדירים environment variable?

ב-Terminal: export ANTHROPIC_API_KEY="sk-ant-..."
או תוסיף לקובץ .env ותשתמש בספרייה python-dotenv.
לא לשים את ה-.env ב-git commit!

Error Handling — מה קורה כשדברים נשברים

1 שעה

למה צריך error handling?

ב-API calls דברים נשברים כל הזמן. אין אינטרנט, מפתח פג תוקף, השרת עמוס, הגעת ל-rate limit. בלי error handling — הקוד קורס עם traceback ומפסיק לרוץ לגמרי. עם error handling — הקוד מטפל בבעיה בצורה נקייה.

מה זה Exception?

Exception (חריגה) היא שגיאה בזמן ריצה. Python "זורק" exception כשמשהו משתבש — חיבור נפל, קובץ לא קיים, מפתח לא נמצא. ה-try/except מאפשר לתפוס את ה-exception לפני שהוא קורס את התוכנית.

try / except — הסינטקס הבסיסי

try_except_demo.py 
import requests

try:
    # קוד שעלול להיכשל
    response = requests.get("https://api.github.com", timeout=5)
    response.raise_for_status()   # זורק exception אם status code >= 400
    data = response.json()
    print(data)

except requests.exceptions.Timeout:
    print("פסק זמן — השרת לא ענה תוך 5 שניות")

except requests.exceptions.HTTPError as e:
    print(f"שגיאת HTTP: {e}")

except requests.exceptions.ConnectionError:
    print("שגיאת חיבור — בדוק אינטרנט")

except Exception as e:
    # תופס כל שגיאה אחרת שלא ציפינו לה
    print(f"שגיאה לא צפויה: {e}")

שים לב ל-response.raise_for_status() — זה כלי חשוב: אם ה-status code הוא 4xx או 5xx, הוא אוטומטית זורק HTTPError. בלעדיו תצטרך לבדוק בעצמך.

מתי לתפוס exception ומתי לא?

when_to_catch.py 
# לא טוב — בולע שגיאות בשקט
try:
    data = call_api()
except:
    pass    # הכי גרוע שאפשר — שגיאה נעלמת בשקט

# לא טוב — תופס יותר מדי
try:
    data = call_api()
except Exception:
    print("משהו נשבר")    # אין מידע מה קרה

# טוב — תופס ספציפי ומטפל נכון
try:
    data = call_api()
except requests.exceptions.Timeout:
    print("timeout — נסה שוב בעוד כמה שניות")
    data = None
except requests.exceptions.HTTPError as e:
    if e.response.status_code == 429:
        print("rate limit — המתן לפני הניסיון הבא")
    raise    # שגיאות אחרות — זרוק הלאה
הכלל הפשוט

תפוס exception רק כשאתה יודע מה לעשות עם השגיאה הספציפית הזאת. אחרת — תן לקוד לקרוס בצורה ברורה. traceback גלוי עדיף על שגיאה בלועה.

retry פשוט עם try/except

retry_example.py 
import requests
import time

def call_api_with_retry(url, max_tries=3):
    """קורא ל-API עם ניסיון חוזר אם יש timeout או rate limit."""
    for attempt in range(1, max_tries + 1):
        try:
            response = requests.get(url, timeout=10)
            response.raise_for_status()
            return response.json()
        except requests.exceptions.Timeout:
            print(f"Timeout בניסיון {attempt}/{max_tries}")
            if attempt == max_tries:
                raise    # הניסיון האחרון — זרוק הלאה
        except requests.exceptions.HTTPError as e:
            if response.status_code == 429:
                wait = attempt * 2
                print(f"Rate limit — מחכה {wait} שניות...")
                time.sleep(wait)
            else:
                raise    # שגיאת HTTP אחרת — זרוק הלאה

# שימוש:
data = call_api_with_retry("https://api.github.com")

Git Workflow היומי

30 דקות
רענון מ-Lessons 1-2

ב-Lessons 1 ו-2 עשינו Git commit + push. עכשיו נלמד את הרוטינה המלאה וגם branching בסיסי.

הפקודות שתשתמש בהן יום-יום

Terminal — Git daily workflow 
git status                              # מה השתנה?
git add lesson3-exercise.py             # הוסף לstaging
git commit -m "Lesson 3: API + error handling"  # commit עם הודעה
git push                                # שלח ל-GitHub

הסדר הוא תמיד: status → add → commit → push.

git pull — לפני שמתחילים לעבוד

Terminal 
git pull        # מושך שינויים מ-GitHub למחשב שלך

חשוב לעשות git pull בתחילת כל עבודה, במיוחד אם עובדים ממחשבים שונים. זה מונע conflicts.

מה זה commit message טוב?

commit message טוב עונה על: מה השתנה ולמה?

Terminal — commit messages 
# לא טוב — לא אומר כלום
git commit -m "update"
git commit -m "fix"
git commit -m "asdf"

# טוב — ברור מה ולמה
git commit -m "Lesson 3: first Claude API call works"
git commit -m "Add try/except for timeout and 429 errors"
git commit -m "Fix: load API key from os.environ, not hardcode"
נוסחה פשוטה

Verb + מה + (למה). לדוגמה: "Add error handling for rate limit" או "Fix: Claude API key not loading from env".

Branch בסיסי — ענפים של קוד

Branch הוא גרסה מקבילה של הקוד. מאפשר לעבוד על פיצ'ר חדש בלי לשבור את main. כשמסיימים — ממזגים חזרה.

Terminal — branching 
# יצירת ענף חדש מ-main
git checkout -b feature/lesson3-exercise

# עבוד על הקוד, עשה commits...
git add .
git commit -m "Lesson 3 exercise: Claude summary script"

# חזרה ל-main
git checkout main

# מיזוג הענף
git merge feature/lesson3-exercise

בשלב הזה תעבוד ישירות על main — branching יהיה חשוב יותר ב-Phase 2 כשתעבוד על פרויקטים גדולים יותר.

ב-GitHub יש מושג נוסף — Pull Request (PR): בקשה רשמית למזג ענף לתוך main. זה גם המקום ל-code review לפני המיזוג.

מה לעולם לא לעשות

קבצים שאסור לעשות להם commit

בדוק שה-.gitignore שלך כולל את הקבצים האלה:

.gitignore

*.pyc               # קבצים מקומפלים
.DS_Store           # קובץ מטא של macOS

אם בטעות עשית commit למפתח API — החלף את המפתח מיד ב-Anthropic Console. נחשב גנוב גם אם ה-repo פרטי — bots סורקים GitHub כל הזמן.
